1 Vision & contexte

LISA (Lecture Intelligente et Structuration Automatisée) est un agent autonome qui transforme les factures douanières arrivant dans une Inbox Drive en JSON structurés au format SYDONIA — sans intervention humaine sauf cas exceptionnel.

Principe directeur

Décision actée

L'IA construit l'usine, la machine tourne.

LISA utilise des scripts Python déterministes par fournisseur chaque fois que possible (coût quasi-nul, latence faible, traçabilité parfaite). Elle n'invoque l'IA qu'aux moments critiques : Vision pour split + détection fournisseur, Opus 4.7 pour générer / réparer les scripts, Gemini V6.1 en filet de sécurité.

Périmètre métier

  • Entrée : factures douanières fournisseurs (PDF natifs ou scans), arrivées via mail/courrier et déposées par les déclarants SIFA dans la Drive Inbox de LISA.
  • Sortie : JSON structuré au schéma SYDONIA V6.1 (header + lines), poussé dans Drive Outbox pour ingestion EDI SIFA.
  • Garantie absolue : LISA produit TOUJOURS un résultat (fallback Gemini si tout le reste échoue), sauf cas explicite de quarantaine (cf. §13).
  • Hors périmètre : déclaration douanière elle-même, signature électronique SYDONIA, gestion comptable.

5 Étape Sanitize (sécurité PDF)

Avant TOUTE étape d'analyse (Vision split, extraction, etc.), le PDF entrant subit un sanitize obligatoire. C'est une étape de défense en profondeur : le PDF vient d'une Inbox Drive accessible aux déclarants SIFA et potentiellement à des envoyeurs externes par rebond mail. On ne le donne pas à un LLM ou à un script Python tant qu'il n'est pas désinfecté.

Pourquoi cette étape

Risque PDF brutMitigation sanitize
JavaScript embarqué (PDF JS) qui pourrait exploiter PyMuPDF ou un rendererqpdf --decrypt --object-streams=generate --remove-restrictions + suppression des actions JS
Formulaires AcroForm avec actions cachéesqpdf aplatit le PDF en sortie
Métadonnées EXIF / XMP fuitant des infos sensibles (auteur réel, GPS, etc.)exiftool -all= -overwrite_original strip tout
PDF chiffré / password-protected qui bloquerait l'extractionqpdf --decrypt retire le password si vide ; sinon escalade en quarantaine "encrypted"
PDF malformé (structure cassée) qui ferait planter PyMuPDFqpdf --linearize reconstruit une structure propre, échoue clairement si réparation impossible
Bombes PDF (taille décompressée énorme, nested streams)Cap taille fichier 50 MB ; cap pages 100 ; qpdf détecte les anomalies de structure
Polices embarquées avec exploits de parserqpdf normalise les flux de polices

Pipeline exact (3 passes)

  INPUT  : /opt/lisa/inbox/<name>.pdf  (PDF brut depuis Drive)
                              │
                              ▼
  ┌────────────────────────────────────────────────────────┐
  │  PASSE 1 — Garde-fous bornes                           │
  │  - taille fichier ≤ 50 MB                              │
  │  - nombre pages ≤ 100 (heuristique fitz.open)          │
  │  - SI dépassement → quarantine "oversized"             │
  └────────────────────┬───────────────────────────────────┘
                       │
                       ▼
  ┌────────────────────────────────────────────────────────┐
  │  PASSE 2 — qpdf                                        │
  │  qpdf --decrypt --object-streams=generate              │
  │       --remove-restrictions                            │
  │       --linearize                                      │
  │       <input> <tmp1>                                   │
  │  - SI exit code ≠ 0 ET pas password → quarantine       │
  │    "qpdf_repair_failed"                                │
  │  - SI password set → tente vide, sinon quarantine      │
  │    "encrypted_unknown_password"                        │
  └────────────────────┬───────────────────────────────────┘
                       │
                       ▼
  ┌────────────────────────────────────────────────────────┐
  │  PASSE 3 — exiftool                                    │
  │  exiftool -all= -overwrite_original <tmp1>             │
  │  Retire toutes les métadonnées EXIF, XMP, IPTC, GPS    │
  └────────────────────┬───────────────────────────────────┘
                       │
                       ▼

           + sha256 du PDF safe (pour idempotence + audit)

Commandes exactes

#!/usr/bin/env bash
# Pseudocode équivalent au module Python lisa_pipeline/sanitize.py

INPUT="/opt/lisa/inbox/${FILENAME}"
TMP1=$(mktemp --suffix=.pdf)
OUTPUT="/opt/lisa/processing/${FILENAME}"

# Passe 2 : qpdf
qpdf   --decrypt   --object-streams=generate   --remove-restrictions   --linearize   --warning-exit-0   "${INPUT}" "${TMP1}"   || { echo "qpdf_repair_failed"; exit 2; }

# Passe 3 : exiftool
exiftool -all= -overwrite_original "${TMP1}"   || { echo "exiftool_failed"; exit 3; }

mv "${TMP1}" "${OUTPUT}"
sha256sum "${OUTPUT}"

Cas d'échec → quarantaine technique

CodeCauseAction
oversizedFichier > 50 MB ou pages > 100Quarantine + alerte Telegram. Possible vraie facture massive ou bombe PDF — Kevin tranche.
encrypted_unknown_passwordPDF chiffré avec password non videQuarantine + alerte. Le déclarant doit fournir un PDF non chiffré.
qpdf_repair_failedStructure PDF corrompue irréparableQuarantine + alerte. Demande de réémission.
exiftool_failedTrès rare (binaire absent ou fichier en lecture seule)Quarantine + alerte technique à Kevin (problème infra, pas métier)

Validation après sanitize

Avant de passer le PDF à Vision Flash, LISA vérifie :

  • fitz.open(output_pdf) ne lève pas d'exception
  • page_count ≥ 1
  • get_text("text", clip=fitz.INFINITE_RECT()) sur p.1 retourne un string (même vide, mais pas d'erreur)
  • Pas de mot-clé JavaScript résiduel dans le contenu brut (/JS, /JavaScript, /AA, /OpenAction) — si présent, quarantine "js_residual" malgré qpdf

Idempotence et déduplication

Le sha256 du PDF safe est calculé et stocké dans la queue. Si le même hash apparaît à nouveau dans l'Inbox (re-dépôt accidentel par un déclarant), LISA détecte le doublon et n'extrait qu'une fois. Le 2e dépôt est archivé directement avec un tag duplicate_of: <original_id>.

Implémentation

Module Python : lisa_pipeline/sanitize.py (existe déjà, à enrichir avec les passes 1 et 3 si pas complètes). CLI : python3 -m lisa_pipeline sanitize <input> <output>. Sortie JSON :

{
  "success": true,
  "output_path": "/opt/lisa/processing/<name>.pdf",
  "sha256": "abc123...",
  "page_count": 4,
  "file_size_bytes": 234567,
  "duplicate_of": null,
  "error_code": null
}

Place dans le flow global

Le sanitize est la 2e étape après la sortie de queue, juste avant l'envoi à Vision Flash. Le diagramme global (§4) le matérialise. Tout PDF qui sort de la queue passe par sanitize sans exception — même les PDF natifs apparemment "propres", car la défense en profondeur ne dépend pas du type de document.

Décision actée

Le sanitize est obligatoire et systématique, jamais court-circuité, même pour les fournisseurs récurrents. Coût négligeable (~50 ms par PDF), bénéfice sécurité énorme. La seule optimisation : si sha256 déjà connu (déduplication), on saute directement à la décision "duplicate_of" sans rejouer le sanitize.

6 Étape Vision split (Gemini Flash)

Pourquoi cette étape

~10 % des PDF entrants contiennent plusieurs factures de fournisseurs différents (mail groupé, scan en bloc, dossier de transit). Avant toute extraction structurée, LISA doit :

  1. Détecter si c'est bien une facture (sinon quarantaine)
  2. Compter le nombre de factures distinctes dans le document
  3. Identifier les fournisseurs et numéros de chaque facture
  4. Vérifier la complétude (manque dernière page → quarantaine)
  5. Splitter en N PDF nommés FOURNISSEUR_N°FACTURE.pdf

Choix Gemini 3 Flash (vs Pro)

Décision actée

Gemini 3 Flash Preview suffit pour cette tâche : analyse visuelle haut niveau sans extraction fine. Coût ~10× inférieur à Gemini 3.1 Pro. Modèle : gemini-3-flash-preview via Vertex AI (service account lisa-496301).

Prompt système (1 seul appel par PDF)

Tu analyses un PDF qui peut contenir une ou plusieurs factures commerciales.

Pour ce document, indique :
- Le nombre de factures distinctes
- Pour chaque facture : nom fournisseur, numéro de facture, plage de pages, complétude

Réponse au format strict, une ligne par facture :
<FOURNISSEUR>_<N°FACTURE> (COMPLETE|INCOMPLETE) Page <X> à <Y>

Règles :
- FOURNISSEUR en MAJUSCULES, espaces remplacés par _, accents retirés
- N°FACTURE : tel qu'imprimé, caractères spéciaux (/, -, .) préservés
- COMPLETE = total HT lisible + au moins 1 ligne produit + cohérence pagination
- INCOMPLETE = manque total / manque dernière page / pas de ligne produit
- Si le document n'est PAS une facture : réponds uniquement NOT_INVOICE
- Si le document est une facture mais sans aucun produit détaillé : NO_PRODUCTS

Exemples :
  BERTRAND_EXPORT_FAC65447 (COMPLETE) Page 1 à 4
  AMAZON_AU_65425-001 (INCOMPLETE) Page 6 à 8

7 Identité fournisseur autonome

Le nom du fournisseur extrait par Vision peut varier entre factures (typos OCR, raison sociale changeante, suffixes sociaux). LISA dédoublonne automatiquement, sans intervention humaine sauf cas ambigu.

Stratégie en 3 étages

  ┌─────────────────────────────────────────────────────────┐
  │  ÉTAGE 1 — SLUG DÉTERMINISTE                            │
  │  • lowercase                                            │
  │  • retire suffixes sociaux (SARL, SAS, LTD, GmbH, …)    │
  │  • retire ponctuation (. , ' ")                         │
  │  • retire accents (NFD strip combining marks)           │
  │  • espaces → _                                          │
  │  Ex : "Citroën Automobiles S.A." → "citroen_automobiles"│

                           │
                           ▼
  ┌─────────────────────────────────────────────────────────┐
  │  ÉTAGE 2 — EMBEDDING SQLITE-VEC                         │
  │  • Calcul embedding du nom normalisé via Haiku/local    │
  │  • Compare aux fournisseurs existants du catalogue      │
  │  • Cosine similarity                                    │
  └────────────────────────┬────────────────────────────────┘
                           │
            ┌──────────────┼──────────────┐

9 Types de documents

Avant de chercher le bon script, LISA classifie automatiquement le type de document : natif, scan propre, scan difficile. C'est ce qui détermine la pile OCR appliquée et le script à utiliser.

Critères de classification

TypeCritère techniqueOCR appliqué
natif PyMuPDF extract_text() retourne > 500 caractères avec layout cohérent (au moins 3 lignes, présence de chiffres) Aucun OCR — texte natif PyMuPDF + PyMuPDF4LLM pour structure
scan_propre PyMuPDF retourne ≤ 500 chars, ET Tesseract avg_confidence ≥ 75 % sur la page de couverture Tesseract --psm 6 --oem 1 --lang fra+eng
scan_difficile PyMuPDF retourne ≤ 500 chars, ET Tesseract avg_confidence < 75 % OpenCV preprocessing (deskew + contrast boost) + Tesseract retry. Si toujours < 60 %, escalade fallback Gemini
Curseurs ajustables

Les seuils 500 chars et 75 % de confiance Tesseract sont des valeurs de démarrage. À ajuster après les premiers retours prod selon le taux de mauvaise classification.

Pré-traitement spécifique scan_difficile

Quand un document est classé scan_difficile, le module script_runner.py applique avant tout OCR :

  1. Conversion en image par page (PyMuPDF, 300 DPI)
  2. Deskew via cv2.minAreaRect sur les contours principaux
  3. Binarisation adaptative (Otsu si contraste OK, sinon Sauvola)
  4. Débruitage léger (cv2.fastNlMeansDenoising)
  5. Tesseract avec --psm 6 --oem 1 --lang fra+eng
  6. Score de confiance global — si toujours < 60 %, on n'essaie même pas le script, escalade directe Gemini

Réutilisation du pré-traitement (Phase 2)

Phase 2 — pas dans la livraison 3

Si LISA détecte qu'un chaînage de pré-traitement spécifique tourne avec succès >3 fois pour un fournisseur, elle pourra (en phase 2) matérialiser ce chaînage en pipeline réutilisable dans /opt/lisa/pipelines/auto_generated/. La livraison 3 reste sur le pipeline figé décrit ci-dessus.

10 Logique de réparation des scripts

Principe simple

Décision actée

LISA ne fait pas de "calibration" générique. Elle répare des scripts existants. Le seul cas où elle crée un script from-scratch est la 1ère facture jamais vue d'un fournisseur (seed Opus sur 1 sample).

Pas de quota journalier de réparations. Le coût Opus est compensé largement par les économies sur les ~95 % de factures qui passent par script déterministe.

Algorithme

INPUT  : facture courante F + slug fournisseur + type t

1. Charger script_actuel = catalogue[slug].methods[t].script
2. Appliquer script_actuel(F) → invoice_json
3. Si validate_math(invoice_json) == OK :
       → drive-push(invoice_json), END
4. Sinon :
   a. Tester les AUTRES scripts du même fournisseur (types ≠ t)
      → s'il y en a un qui marche, on bascule (probable erreur de classify)
   b. Si aucun ne marche :
      → Charger 4 factures précédentes du même (slug, t) depuis samples/
      → samples_set = [F, F-1, F-2, F-3, F-4]
      → Appeler Opus 4.7 avec :
          * script_actuel
          * samples_set
          * grimoire.query(supplier=slug, category="business_rule")
          * grimoire.query(supplier=slug, category="extraction_quirk")
          * Prompt "Génère un script déterministe qui marche pour ces 5 factures"
      → script_candidat = Opus.generate()
      → Tester script_candidat sur samples_set ENTIER
          * Si OK sur 5/5 :
              - script_actuel → version archivée
              - script_candidat → catalogue[slug].methods[t].script v_n+1
              - commit Git, capitalisation grimoire
              - drive-push(script_candidat(F))
              - END
          * Si KO sur ≥ 1 :
              → FALLBACK GEMINI V6.1, capitalisation échec dans grimoire

Cas particulier : 1ère facture d'un fournisseur nouveau

INPUT  : facture F + slug = "nouveau_fournisseur" (créé à l'instant)

1. catalogue[slug].methods[t] == null
2. Appeler Opus 4.7 SEED avec :
     * samples_set = [F]   (1 seule)
     * Prompt "Génère un script seed pour cette facture, robuste aux variations"
3. script_seed = Opus.generate()
4. Tester sur F seul → si OK, drive-push, sample stocké
5. Si KO → FALLBACK GEMINI (1ère facture, pas de capitalisation suffisante)

   → Facture 2 du même fournisseur : on a maintenant 2 samples,
     on applique script_seed, si KO → Opus reprend sur 2 samples,
     etc.

Contenu du prompt Opus 4.7 (squelette)

SYSTEM :
Tu es un générateur expert de scripts Python d'extraction de factures
douanières au schéma SYDONIA V6.1. Tu produis du code déterministe utilisant
des positions strictes, des regex, des positions de blocs de mots en abscisse
et ordonnée. Tu utilises l'OCR la plus adaptée au type de document.

CONTRAINTES :
- Le script DOIT marcher sur les 5 factures samples fournies (math + schéma)
- Imports allowlistés : pymupdf, cv2, pytesseract, pdfminer, pandas, re, pathlib
- Pas de os.system, requests, eval, exec
- Output strict : un dict Python conforme au schéma V6.1 (header + lines)

INPUT :
- script_actuel (Python, version v_n)
- samples_set : 5 PDF + leur JSON golden attendu
- business_rules pour ce fournisseur (issues du grimoire) :
    "Les produits alimentaires type saucisson : qty en mètres linéaires"
- extraction_quirks pour ce fournisseur :
    "Le total HT est précédé par 'Net to pay' et non 'Total HT'"

OUTPUT :
Un script Python complet entre balises <script>...</script>
Caching pour Opus

2 breakpoints cache_control : (1) sur le SYSTEM prompt stable, (2) sur la 5e facture (rolling window — change uniquement quand nouvelle facture arrive). TTL 1h. Économise ~85 % du coût input Opus.

11 Health score décroissant

Principe

Un script n'est jamais définitivement scellé. Sa fiabilité est mesurée en continu sur les 5 dernières factures. Si elle décroche, LISA relance automatiquement la réparation.

Calcul

health_score = somme(confidence_i pour i in rolling_window) / 5

où confidence_i ∈ [0.0, 1.0] selon le résultat de la facture i :
  outcome == "ok"        → confidence = 1.00
  outcome == "partial"   → confidence = 0.50 + 0.50 × (lines_extracted / lines_expected)
  outcome == "fail"      → confidence = 0.00
  outcome == "quarantine"→ ignoré dans le calcul (n'entre pas dans la window)

Seuils de décision

Plage healthÉtatAction
≥ 0.95ScelléPas de nouvelle réparation. Script utilisé en confiance.
0.80 — 0.94AcceptableScript utilisé. Pas de réparation automatique mais alerte si une facture casse.
0.60 — 0.79Drift mineurRéparation Opus à la prochaine facture qui échoue.
< 0.60Drift majeurRéparation Opus immédiate, on ne tente même pas la facture courante avec le script existant.

Rolling window FIFO 5

À chaque nouvelle facture traitée :

  1. L'entrée la plus ancienne sort de la window
  2. La nouvelle entrée entre en queue
  3. health_score est recalculé
  4. Si seuil franchi → action correspondante déclenchée

Que reste-t-il des anciennes factures hors window ?

  • PDF original : déjà déplacé vers Drive ARCHIVE après traitement initial. Pas conservé en local.
  • JSON extrait validé : commit Git du catalogue (push quotidien 23h). Restauration possible si besoin.
  • Métadonnées : history.yaml conserve une trace courte (date, facture, outcome) au-delà de la window pour audit.
Implémentation

Le health_score est calculé dans lisa_pipeline/catalogue.py, fonction compute_health(slug, type). Persisté dans meta.yaml après chaque mise à jour.

Thinking protocol obligatoire (6 étapes)

  1. Contextualisation globale (compter factures, pays expéditeur, mention DOF)
  2. Extraction ligne par ligne (cascade ref / origin / classification)
  3. Résolution origine EU/UE (jamais "EU" final)

Quand Gemini réussit là où un script déterministe échoue, LISA :

  1. Pousse le JSON Gemini vers Outbox immédiatement
  2. Stocke la facture dans samples/<type>/ comme golden
  3. Si on a maintenant ≥ 2 samples du même fournisseur/type avec Gemini → tente une nouvelle génération Opus avec ces samples (peut-être qu'avec 2-3 exemples Gemini, Opus arrive à faire un script déterministe)
  4. Ajoute leçon grimoire catégorie extraction_quirk : "Fournisseur X type Y : Gemini réussit où Opus échoue, particularités layout..."
Implémentation

Module : lisa_pipeline/gemini_fallback.py. CLI : python3 -m lisa_pipeline gemini-fallback <pdf>. Sortie : JSON SYDONIA + tokens_in / tokens_out / cached_tokens / duration_s.

13 Grimoire RAG (3 catégories)

Le grimoire est une mémoire vectorielle (sqlite-vec) qui capitalise toutes les leçons apprises au fil de l'eau. Sonnet l'interroge avant chaque décision difficile, Opus le consulte pendant la génération de script.

Schéma de leçon

id              : uuid
supplier_slug   : "bertrand_export" (ou "*" pour règles transverses)
type            : "natif" | "scan_propre" | "scan_difficile" | "*"
category        : "extraction_quirk" | "ocr_method" | "business_rule"
title           : string court
content         : string libre, en français
embedding       : vector(384)
created_at      : ISO 8601
created_by      : "auto" | "declarant.alice" | "kevin"
active          : bool
usage_count     : int (incrémenté à chaque query qui retourne cette leçon)

Les 3 catégories

1. extraction_quirk (auto)

Particularités layout / patterns / pièges d'extraction découverts en cours de route. Ajoutés automatiquement par l'agent Sonnet ou Opus quand ils résolvent un cas non trivial.

Exemples :

  • "Pour BERTRAND EXPORT, le total HT est précédé par 'Net to pay' (anglais), pas 'Total HT' (français)."
  • "Sur les factures Citroën, la colonne origine peut contenir 'EU' — résoudre via adresse expéditeur (FR)."
  • "Les références produit du fournisseur ABC contiennent des slashes (WF-2810/30/35/50DWF) qu'il faut préserver intacts."

2. ocr_method (auto)

Méthodes OCR ou pré-traitement qui ont marché là où les défauts échouaient.

Exemples :

  • "ZHANG INDUSTRIES, scan difficile : Tesseract --psm 6 + lang chi_sim+eng bat RapidOCR sur scans CJK basse résolution."
  • "AUTO PIECES, scan : deskew + binarisation Sauvola (pas Otsu) car contraste très faible."

3. business_rule (manuel via GAS)

Règles métier saisies par les déclarants SIFA via la vue "Instructions" du dashboard GAS. Elles sont injectées dans le prompt Opus lors de la génération de script ET dans le system prompt Sonnet quand il traite ce fournisseur.

Exemples :

  • "Les produits alimentaires type saucisson : quantité exprimée en mètres linéaires, pas en unités."
  • "Pour ce fournisseur, le code HS dans la colonne 7 est toujours plus fiable que celui en colonne 4."
  • "Devise par défaut si non précisée sur les factures FR : EUR. Si non précisée sur AU : AUD."

Query patterns

# Avant calibration Opus : récupérer les leçons utiles
context = grimoire.query(
    supplier_slug="bertrand_export",
    type="scan_propre",
    categories=["extraction_quirk", "business_rule", "ocr_method"],
    top_k=10,
    min_similarity=0.7,
)

# Avant orchestration Sonnet : règles transverses
context_global = grimoire.query(
    supplier_slug="*",
    categories=["business_rule"],
    top_k=5,
)

Maintenance

  • Désactivation : Kevin peut désactiver une leçon obsolète via python3 -m lisa_pipeline grimoire-deactivate <id> ou via GAS
  • Cleanup : leçons active: false avec usage_count == 0 depuis 6 mois → archivage Git puis suppression
  • Backup : grimoire.db inclus dans le push Git quotidien
Implémentation

Module : lisa_pipeline/grimoire.py. Schéma sqlite-vec : table lessons + index vectoriel lessons_vec. Embeddings via Haiku 4.5 (model d'embedding séparé Anthropic) ou modèle local si Haiku embedding pas dispo.

14 Quarantaine

La quarantaine est un cas d'exception. LISA produit toujours un résultat sauf quand le document n'est pas une facture exploitable. Trois cas :

CasDétectionAction
Pas une factureVision Flash retourne NOT_INVOICEDéplace PDF vers /opt/lisa/quarantine/not_invoice/ + alerte Telegram
Pas de produits détaillésVision Flash retourne NO_PRODUCTSDéplace PDF vers /opt/lisa/quarantine/no_products/ + alerte Telegram
Facture incomplèteVision Flash retourne (INCOMPLETE) sur au moins une factureDéplace PDF vers /opt/lisa/quarantine/incomplete/ + alerte Telegram. Si multi-factures, les COMPLETE continuent le flow.

Format alerte Telegram

⚠ QUARANTAINE — BERTRAND_EXPORT_FAC65447.pdf
Raison : INCOMPLETE (manque la dernière page selon Vision Flash)
Pages : 1 à 4 (le total HT n'est pas lisible)
Action : vérifier le scan ou demander réémission au fournisseur.

Résolution manuelle (vue GAS "Résolution")

Kevin (ou un déclarant SIFA via GAS) peut :

  • Force-process : forcer LISA à traiter quand même la facture, malgré l'alerte Vision (utile si Vision se trompe)
  • Reject définitif : marquer la facture comme rejetée, déplace vers quarantine/rejected/, l'archive existe mais le déclarant SIFA sait qu'il doit demander une réémission
  • Upload de remplacement : déposer le PDF complet via la vue GAS, LISA reprend le flow normal sur le nouveau fichier

Cas non couvert par la quarantaine

Quand un document est une facture complète, mais que LISA n'arrive pas à l'extraire correctement (Vision + script + Gemini tous échouent) → pas de quarantaine. LISA produit un JSON marqué incomplet et alerte Kevin. Le déclarant doit corriger manuellement via GAS. Ce cas devrait être très rare avec le V6.1 en filet.

hit_ratio = u.cache_read_input_tokens / max(total_input, 1)
logger.info(f"sonnet hit_ratio={hit_ratio:.2%}")

# Implicit caching automatique si prompt byte-stable

Pièges à éviter

Pièges qui cassent le caching
  • Timestamps dans le system prompt → cache invalide à chaque seconde. Mettre la date dans le user message.
  • Ordre des clés JSON non stable (Swift, Go) → vérifier le SDK.
  • Concurrent requests avant 1re réponse → cache pas encore créé, attendre.
  • Anthropic via Vertex AI → pas d'automatic caching, pas de workspace isolation. Toujours via Claude API directe.
  • Prompt chargé via Markdown re-rendu → byte-différent. Charger via open(..., "rb").
